## Choosing the right job template

When selecting an AWX job template to run manually, prefer templates that have
`teams:` permissions defined over templates that have `schedules:` blocks.
Templates with schedules are designed for automated runs. Templates with team
permissions are the intended entry point for on-demand use. Both may have the
same underlying playbook and tags -- the distinction is about intent and
operational context.

## Minimum necessary execution (MANDATORY)

Before launching ANY AWX job:

1. **Find the role's available tags.** Check the role's `tasks/main.yml` for
   `tags:` on each task block. Identify which tag(s) cover ONLY the change
   being applied.
2. **Pass `--tags` to limit execution.** Never run a full role when a targeted
   tag exists. A compose template + container restart should take seconds, not
   30 minutes of vault lookups and minio provisioning.
3. **Example:** To apply a docker-compose config change to Kafka, use
   `--tags start_broker` (templates compose, restarts container, validates).
   Do NOT run the full role which includes cert rotation, ACLs, ES index
   setup, and firewall rules.
4. **If unsure which tag to use:** Read the role tasks file FIRST. Do not
   guess. Do not launch without tags "just to be safe." The safe option is
   the minimal one.

This applies to ALL job launches: Looker, Kafka, Redis, application deploys,
everything. Full-role runs are for maintenance windows only.

## Deriving the correct limit

When specifying a `limit` for an AWX job, derive it from the inventory source's
`keyed_groups` configuration (e.g., `inventory_aws_phi/prd.aws_ec2.yml`), not
from existing schedule definitions in `job_templates.yml`. Schedule `limit`
values may use shorthand aliases that differ from the actual inventory group
names generated by the dynamic inventory.

## Using tachi for AWX

Prefer `tachi awx` commands over raw API calls or the AWX UI for on-demand job
launches. tachi provides audit-friendly invocations and handles profile
selection automatically.

**Use the template catalog** before running jobs. The catalog provides:
- Alias resolution (short names for long template names)
- Known-good limit values per template
- Survey field documentation (required params, choices, defaults)
- Run history (what params worked last time)

```
tachi awx catalog search <query> --profile phi   # find templates
tachi awx catalog show <name-or-alias> -p phi    # full detail
tachi awx jobs run --template <alias> -p phi     # aliases resolve automatically
```

Common commands:
```
tachi awx jobs run --template "Deploy Channels" --profile phi
tachi awx jobs run --template restart --profile phi --limit aws_uat
tachi awx jobs search --template "cert" --status failed
tachi awx jobs get <job-id> --detail
```
